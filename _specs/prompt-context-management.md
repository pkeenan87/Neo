# Spec for Prompt Context Management

branch: claude/feature/prompt-context-management

## Summary

The agent frequently exceeds Claude's 200K token input limit during normal usage. A session with just 3–4 `run_sentinel_kql` tool calls can push the prompt past 200K tokens, returning a `400 invalid_request_error`. This happens because conversation messages (including full tool call results) accumulate unbounded in the `localMessages` array and are sent verbatim to the Claude API on every turn.

This feature introduces context window management to keep prompts within token limits while preserving the agent's ability to reason over prior findings.

## Functional requirements

- Count (or estimate) the token size of the full message array before each Claude API call
- When the estimated token count approaches a configurable threshold (e.g. 180K tokens), automatically reduce the context size before sending
- Truncate or summarise large tool results (especially KQL query responses) at ingestion time so they don't bloat the context
- Preserve the most recent messages and the system prompt in full — only compress or drop older middle-of-conversation content
- Apply a per-tool-result size cap so that a single KQL response cannot consume an outsized share of the context window
- Log or surface a warning when context trimming occurs so the user (and developers) know information was condensed
- Ensure the agent can still reference key earlier findings after trimming (via a rolling summary or pinned facts)

## Possible Edge Cases

- A single tool result that alone exceeds the per-result cap (e.g. a KQL query returning thousands of rows) — must be truncated at ingestion, not just at send time
- Rapid back-to-back tool calls in one agent loop iteration that collectively blow past the limit before the next check
- Summarisation itself failing or returning an unhelpful summary — need a fallback (e.g. hard truncation with a note)
- Sessions resumed from Cosmos DB that already contain a large message history — context management must apply on resume, not just during live sessions
- The system prompt + latest user message alone approaching the limit — should return a clear error rather than silently dropping the user's input

## Acceptance Criteria

- A session with 5+ KQL tool calls completes without hitting the 200K token error
- Individual tool results exceeding the per-result cap are truncated with a note indicating truncation
- When context trimming activates, a visible indicator or log message is emitted
- The agent can still reference key findings from earlier in the conversation after trimming occurs
- No regression in agent quality for short conversations that stay well within limits
- Both CLI and web paths are covered (the shared agent loop in `web/lib/agent.ts` handles both)

## Open Questions

- What token estimation approach to use? Options: (a) a lightweight tokenizer like `tiktoken` / `@anthropic-ai/tokenizer`, (b) character-count heuristic (÷4), (c) the API's `usage` field from prior responses to track cumulative input tokens. The usage field
- Should large tool results be truncated eagerly at ingestion or lazily only when the context is too large? lazily
- What is the right per-tool-result cap? 20K tokens? 50K? Should it be configurable per tool? Lets increase to 50k
- Should the rolling summary be generated by a cheaper/faster model (e.g. Haiku) to avoid latency and cost? yes
- Is there value in letting the agent explicitly request "show me the full result" for a previously truncated tool call (re-fetch from storage)? yes

## Testing Guidelines

Create a test file(s) in the ./test folder for the new feature, and create meaningful tests for the following cases, without going too heavy:

- Token estimation returns a reasonable count for a known message array
- A message array over the threshold is correctly trimmed to fit within the limit
- The system prompt and most recent N messages are always preserved after trimming
- Large tool results are truncated to the configured cap with a truncation notice appended
- Context trimming emits a warning/log when it activates
- Edge case: a single tool result larger than the cap is truncated at ingestion time
