# Prompt Context Management

## Context

The agent hits Claude's 200K token input limit during normal investigations with 3-4 KQL tool calls because `localMessages` in `web/lib/agent.ts` accumulates full tool results unbounded and sends them verbatim on every API call. This plan introduces a new `context-manager.ts` module that (a) estimates token usage via the API's `usage.input_tokens` field, (b) truncates oversized tool results lazily before sending, and (c) compresses older conversation turns via a Haiku-powered rolling summary when the context window is near capacity. A per-tool-result cap of 50K tokens prevents any single response from dominating the context.

---

## Key Design Decisions

- **Token estimation via API `usage` field**: After each Claude response, `usage.input_tokens` tells us exactly how many input tokens were consumed. Track this cumulatively so we know the current context size without needing a tokenizer library. For the very first call (no prior usage data), use a character-count heuristic (chars ÷ 4) as a conservative estimate.
- **Lazy trimming, not eager**: Tool results are stored in full in `localMessages` and in Cosmos DB. Trimming happens only when preparing the messages array for the next API call — this preserves the full data for re-fetch and keeps the storage layer unchanged.
- **50K token per-tool-result cap**: When preparing messages for the API, any single tool result content string exceeding ~200K characters (50K tokens × 4 chars/token) is truncated with a notice. This prevents a single KQL response from consuming a quarter of the context window.
- **Rolling summary via Haiku**: When total context approaches 160K tokens, older middle-of-conversation messages are replaced with a compact summary generated by `claude-haiku-4-5-20251001`. The system prompt, the summary, and the most recent N messages are preserved. If summarisation fails, fall back to hard truncation with a notice.
- **Re-fetch capability**: Add a new tool `get_full_tool_result` that lets the agent request a previously truncated tool result by its `tool_use_id`, retrieved from the full messages stored in the session/Cosmos.

## Implementation Steps

### 1. Add configuration constants to `web/lib/config.ts`

- Add four exported constants at module level: `CONTEXT_TOKEN_LIMIT = 180_000`, `TRIM_TRIGGER_THRESHOLD = 160_000`, `PER_TOOL_RESULT_TOKEN_CAP = 50_000`, `PRESERVED_RECENT_MESSAGES = 10`
- These are the token-count thresholds; the character-based equivalents (multiply by 4) are derived in the context manager

### 2. Add types to `web/lib/types.ts`

- Add `ContextTrimEvent` to the `AgentEvent` union: `{ type: "context_trimmed"; originalTokens: number; trimmedTokens: number; method: "truncation" | "summary" }`
- Add `"context_trimmed"` to the `AgentEventType` union
- Add `GetFullToolResultInput` interface with field `tool_use_id: string`

### 3. Create `web/lib/context-manager.ts`

- **`estimateTokens(messages: Message[]): number`** — Walks the message array, sums character counts of all string content (including serialized tool result blocks), divides by 4. Used only when no prior API usage data is available.
- **`truncateToolResult(content: string, capTokens: number): string`** — If content length exceeds `capTokens * 4`, truncate to that limit and append `\n\n[Result truncated from {original} to {truncated} characters. Use get_full_tool_result with the tool_use_id to retrieve the complete output.]`
- **`prepareMessages(messages: Message[], lastInputTokens: number | null, systemPromptTokenEstimate: number): { messages: Message[]; trimmed: boolean; method?: "truncation" | "summary"; originalTokens: number; newTokens: number }`** — The main entry point called before each API call:
  1. Deep-copy the messages array (do not mutate the originals stored in session)
  2. Walk all tool_result blocks and apply `truncateToolResult()` to any content exceeding the per-result cap
  3. Estimate total tokens: if `lastInputTokens` is available, use it as a baseline adjusted for new messages added since; otherwise use `estimateTokens()`
  4. If estimated total exceeds `TRIM_TRIGGER_THRESHOLD`, invoke `compressOlderMessages()`
  5. Return the prepared messages array and metadata about whether trimming occurred
- **`compressOlderMessages(messages: Message[], preserveCount: number): Promise<Message[]>`** — Takes the full prepared messages array:
  1. Split into three segments: messages[0] (always keep first user message for context), middle messages (candidates for compression), and the last `preserveCount` messages (always preserved)
  2. Call Claude Haiku with the middle messages asking for a 3-5 bullet point summary of the investigation so far, focusing on findings and actions
  3. Replace the middle segment with a single user message containing the summary prefixed with `[Context compressed — earlier investigation summary:]`
  4. If the Haiku call fails, fall back to simply dropping the middle messages and inserting a notice: `[Earlier conversation context was removed to stay within token limits. Key findings may need to be re-investigated.]`

### 4. Update `web/lib/agent.ts` — `runAgentLoop()`

- Import `prepareMessages` from `context-manager`
- Add a local variable `lastInputTokens: number | null = null` before the while loop
- Before each `createWithRetry()` call, call `prepareMessages(localMessages, lastInputTokens, systemPromptTokenEstimate)` where `systemPromptTokenEstimate` is a rough estimate of the system prompt size (calculate once before the loop using character count ÷ 4)
- Pass the prepared (possibly trimmed) messages to `createWithRetry()` instead of raw `localMessages`
- After each successful API response, update `lastInputTokens = response.usage.input_tokens`
- If `prepareMessages` reports trimming occurred, invoke the `onContextTrimmed` callback (new optional callback) so the stream layer can emit the event
- Continue storing full (untrimmed) tool results in `localMessages` — only the API call sees the trimmed version

### 5. Update `web/lib/agent.ts` — `resumeAfterConfirmation()`

- Apply the same `prepareMessages()` pattern: the function calls `runAgentLoop()` which already handles it, but ensure the messages passed in are the full untrimmed messages from the session

### 6. Add `get_full_tool_result` tool

- In `web/lib/tools.ts`: Add a new tool schema with name `get_full_tool_result`, description explaining it retrieves the full content of a previously truncated tool result, and a single required parameter `tool_use_id` (string)
- In `web/lib/executors.ts`: Add executor function that receives the `tool_use_id`, searches backward through the current session's `localMessages` for a `tool_result` block with matching `tool_use_id`, and returns its full content. If not found, return an error message. This executor needs access to the session messages — pass them via a closure or add a session context parameter to `executeTool()`
- In `web/lib/permissions.ts`: Add `get_full_tool_result` to both admin and reader tool lists (it's read-only)

### 7. Update `web/lib/stream.ts` and types

- Handle the new `context_trimmed` event type in `encodeNDJSON` (it already handles any `AgentEvent`, but update the type union)
- Add `onContextTrimmed` to `AgentCallbacks` in `web/lib/types.ts` as an optional callback

### 8. Add error handling for the 400 token limit error

- In `createWithRetry()` in `web/lib/agent.ts`, catch the specific 400 error with message containing "prompt is too long"
- When caught, log a warning and return a user-friendly error message: "The conversation has grown too large. Please start a new session." rather than throwing an opaque error
- This serves as a safety net if the context manager's estimates are off

### 9. Create tests in `test/context-manager.test.ts`

- Set up a test runner (vitest, since the project uses TypeScript and ES modules — add as a dev dependency)
- **Test: `estimateTokens` accuracy** — Create a known message array, verify the estimate is within a reasonable range (not testing exact tokenizer accuracy, just that the heuristic works)
- **Test: `truncateToolResult` with content under cap** — Verify content is returned unchanged
- **Test: `truncateToolResult` with content over cap** — Verify content is truncated and notice is appended
- **Test: `prepareMessages` preserves recent messages** — Build a message array where older messages push past the threshold, verify the last N messages are untouched
- **Test: `prepareMessages` with small conversation** — Verify no trimming occurs when under threshold
- **Test: `prepareMessages` with oversized tool result** — Verify individual tool results are capped even if total context is under threshold
- **Test: fallback when summary fails** — Mock the Haiku call to throw, verify hard truncation fallback is used with appropriate notice
